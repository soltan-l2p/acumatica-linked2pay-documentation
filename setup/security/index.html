Tokenization, credentials, webhooks, roles, and audit

Security and Data Handling

Apply least privilege, protect secrets, authenticate webhooks, define PCI boundaries, and keep sensitive information out of screenshots and support channels.

Tokenization and sensitive data

The integration expects linked2pay contact/payment-profile tokens; raw-card fallback is unsupported. Acumatica stores identifiers, masked descriptors, and transaction IDs—not full card/bank numbers.

Never store or transmit: API keys, full card numbers, CVV, full bank/routing data, unmasked tokens, or customer personal information in screenshots, notes, chat, email, source control, or support tickets.

PCI responsibility boundaries

Tokenization reduces exposure but does not automatically remove all PCI obligations. The merchant retains responsibility for scope determination, access controls, policies, endpoints, and approved payment-entry methods. Consult the merchant's PCI assessor/compliance owner.

API-key storage and rotation

  • Store API and webhook keys only in authorized processing-center settings or approved secret management.
  • Restrict Processing Centers and database exports.
  • Separate sandbox/production credentials.
  • Rotate on schedule and after exposure/staff transition.
  • Coordinate both sides, then retest authentication/webhook.
  • Never place secret values in screenshots or shared exports.

Webhook authentication

The handler requires X-Api-Key and compares it to WEBHOOKKEY on active linked2pay centers. Missing/invalid keys return HTTP 401.

  • HTTPS only; preserve header through proxies.
  • High-entropy environment-specific key.
  • Monitor repeated 401/400/500 responses.
  • Restrict/rotate logs containing transaction identifiers.

Audit logging and retention

Enable Acumatica audit history for customers, methods, centers, and AR payment records. Retain transaction/history, trace, and webhook logs under customer policy. Limit file-system access and mask personal data before escalation.

Sandbox data rules

  • Synthetic identities/payment data only.
  • No production tokens in sandbox.
  • No unmasked production database restores.
  • Expire sandbox credentials after the project.
