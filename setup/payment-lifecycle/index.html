Payment Lifecycle

Understand card and ACH authorization, capture, settlement, declines, voids, refunds, returns, recurring failures, duplicate prevention, and accounting effects.

Accounting state versus gateway state

Important: an Acumatica document being released does not always mean a card or ACH transaction has settled. Review Acumatica document status, approved CCProcTran rows, linked2pay Payment Status/Webhook History, and the linked2pay portal together.

The integration records transaction identifiers, normalized status, request state, and history. A local financial state is marked paid, voided, or refunded only when the corresponding Acumatica processing transaction is approved.

Level 2 and Level 3 processing

Card-brand data is not always supplied by the wallet response, so the integration does not assign a commercial level by assumed card brand.

  • When Level 3 is enabled, the integration attempts Level 3, then Level 2, then standard processing.
  • When only Level 2 is enabled, it attempts Level 2, then standard processing.
  • Fallback occurs only after a definite gateway rejection with no transaction identifiers. A timeout or ambiguous result is held for reconciliation and is never retried automatically at another level.
  • Level 3 requires eligible invoice applications and line-level commercial data. Partial applications or missing commercial data fall back safely.

Credit-card fee behavior

  • When an eligible linked2pay card method is selected and the payment is saved, the configured fee is calculated automatically.
  • The integration creates, releases, and applies one CCFEE debit memo.
  • Changing the payment method to ACH removes an unprocessed fee; changing back to card recalculates it.
  • Remove Processing Fee allows an exception before processing, and Apply Fee can add it again.
  • Fee changes are blocked after authorization, capture, release, or an external status update.

Duplicate prevention and ambiguous outcomes

Each request uses the Acumatica payment reference as an external reference and records request state before contacting linked2pay. Concurrent capture attempts are serialized, and approved results are reused rather than charged again.

Unknown result: if linked2pay may have succeeded but Acumatica did not receive a complete response, the payment is placed on reconciliation hold. Verify the portal and transaction history first. Use Allow linked2pay Retry only after confirming that no gateway transaction exists.

Auto-Bill release recovery

If capture succeeds but Acumatica payment release fails, Auto-Bill does not capture again. The payment appears under Captured But Unreleased on AR508800. Use Retry Captured Payment Releases after correcting the release issue, or export the list for reconciliation.

Webhook behavior

Authenticated webhooks normalize statuses and update linked2pay tracking/history. Duplicate events are acknowledged once, and unmatched or out-of-order events are stored for retry. Portal-originated voids and refunds are marked as external states; complete the matching Acumatica financial action when required.

Reconciliation checklist

  • Acumatica and linked2pay amounts, including card fee, match.
  • Approved processing and gateway transaction IDs are recorded.
  • Status is final for the rail.
  • Cash account matches the payment method and bank settlement.
  • Invoice and payment are released and the zero-balance invoice is closed.
  • Captured-but-unreleased, external void/refund, return, and chargeback exceptions are resolved.
